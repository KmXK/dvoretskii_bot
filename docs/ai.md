# Работа с ИИ

## Архитектура

```
┌─────────────────────────────────────────────────────────────────┐
│                         Handler                                 │
│  (ai_handler, pasha_handler, ...)                               │
│                                                                 │
│  register_ai_handler("name", ai_call)                           │
│  execute_ai_request(context, text, ai_call, "name")             │
└──────────────────────────┬──────────────────────────────────────┘
                           │
              ┌────────────▼────────────┐
              │     ai_context.py       │
              │                         │
              │  build_reply_context()  │──── Telethon: собирает
              │  execute_ai_request()   │     цепочку реплаев
              │  register/get_ai_handler│
              └────────────┬────────────┘
                           │
         ┌─────────────────┼─────────────────┐
         ▼                 ▼                 ▼
   AI-провайдеры (ai.py): Yandex, DeepSeek, OpenRouter
         │                 │                 │
         ▼                 ▼                 ▼
┌────────────────────────────────────────────────────┐
│              AiRelatedMessageHandler                │
│                                                     │
│  Ловит реплаи на сообщения бота, определяет         │
│  провайдера по полю handler в ai_messages,           │
│  продолжает диалог через тот же AI                   │
└────────────────────────────────────────────────────┘
```

## Провайдеры

Определены в `steward/helpers/ai.py`. Каждый провайдер — функция, которая принимает `user_id`, `messages` и `system_prompt`, возвращает строку ответа.

| Провайдер | Функция | Async | Примечание |
|---|---|---|---|
| **Yandex GPT** | `make_yandex_ai_query` | да | Самый дешёвый, используй по умолчанию |
| **OpenRouter** | `make_openrouter_query` | да | Любые модели, ходит через `DOWNLOAD_PROXY` |
| **DeepSeek** | `make_deepseek_query` | нет | Принимает `text` вместо `messages`, не поддерживает контекст реплаев |

Доступные модели и env-ключи — см. классы `YandexModelTypes`, `OpenRouterModel` в `ai.py`.

## Формат сообщений

Yandex и OpenRouter принимают `messages: list[tuple[str, str]]` — список кортежей `(role, text)`:

```python
messages = [
    ("user", "Привет"),
    ("assistant", "Здарова"),
    ("user", "Как дела?"),
]
```

## Промпты

Лежат в `prompts/*.txt`. Загружаются при старте через `get_prompt()` в `ai.py` как константы модуля.

Чтобы добавить новый: создай файл в `prompts/` и добавь в `ai.py`:

```python
MY_PROMPT = get_prompt("имя_файла")
```

### Динамические описания для `/ai`

Для обработчика `/ai` системный промпт собирается динамически:
- берётся шаблон из `prompts/grok_short_aggressive.txt`
- маркер `{{USERS_DESCRIPTIONS}}` заменяется на список из `User.stand_name` + `User.stand_description`
- если пользователей нет, подставляется заглушка

Важно: в промпт эти данные подаются как описания пользователей.

Команда для управления источником этих описаний:
- `/stands` — список всех пользователей
- `/stands add <имя>` — запуск пошагового добавления (описание + пользователь)
- `/stands remove <имя>` — удалить пользователя

## Контекст реплаев

`execute_ai_request` автоматически собирает историю диалога из цепочки реплаев в чате.

### Как это работает

1. `build_reply_context()` берёт текущее сообщение и идёт по цепочке `reply_to` через Telethon
2. Для каждого сообщения определяет роль: от бота — `assistant`, иначе — `user`
3. Если прямого реплая нет, ищет связь через `db.ai_messages` (маппинг ответ бота → исходное сообщение)
4. Собранный контекст передаётся в AI-провайдер

### Хранение в БД

После ответа бота в `db.ai_messages` сохраняется `AiMessage` (`timestamp`, `message_id`, `handler`). Максимум 1000 записей, старые вытесняются.

## Как написать новый AI-хендлер

1. Создай промпт в `prompts/`, загрузи в `ai.py`
2. В хендлере вызови `register_ai_handler` с именем и callable — это включит продолжение диалога через реплаи
3. В методе `chat` вызови `execute_ai_request` с нужным провайдером, промптом и тем же именем
4. Зарегистрируй хендлер в `main.py`

`execute_ai_request` берёт на себя: сборку контекста, вызов AI, отправку ответа, сохранение в БД.

Референсные примеры: `ai_handler.py` (OpenRouter), `pasha_handler.py` (Yandex).

## AI Router (natural language → команда)

`AiRouterHandler` (`steward/handlers/ai_router_handler.py`) позволяет выполнять команды бота на естественном языке.

### Триггеры

Сообщение должно начинаться с одного из:
- `@bot_username`
- `дворецкий`
- `уважаемый`
- ответ (reply) на сообщение бота или пересланное сообщение бота — триггер-слово не нужно

### Примеры

Простая команда:
```
уважаемый, напомни через 10 минут покормить кота
→ 🤖 /remind 10m покормить кота  [✅ Выполнить] [❌ Отмена]
```

Пересланное сообщение + реплай:
```
[Пересланное: "Встреча 25 февраля в 10:00"]
↳ дворецкий, напомни
→ 🤖 /remind 25.02 10:00 Встреча  [✅ Выполнить] [❌ Отмена]
```

Бан по пересланному сообщению:
```
[Пересланное от @spammer: "купи биткоин"]
↳ дворецкий, забань этого человека на 5 минут
→ 🤖 /ban @spammer 5m  [✅ Выполнить] [❌ Отмена]
```

Несколько команд:
```
уважаемый, напомни через час и добавь в список дел: купить молоко
→ 🤖 /remind 1h купить молоко
     /todo купить молоко
  [✅ Выполнить] [❌ Отмена]
```

### Как работает

1. Ловит сообщения начинающиеся с триггера или ответы на сообщения бота (не команды `/`)
2. Если сообщение — реплай, текст и информация об отправителе `reply_to_message` передаётся LLM как контекст (включая `forward_origin` для пересланных сообщений). Таблицы долгов из отчётов `/bill` автоматически преобразуются в текстовый формат «X должен Y: сумма» для корректного парсинга
3. Собирает `help()` всех хендлеров → каталог команд (с исключениями)
4. Отправляет в Yandex GPT: каталог + контекст + запрос пользователя
5. LLM возвращает одну или несколько команд (по одной на строку)
6. Бот показывает команды с кнопками подтверждения
7. По нажатию ✅ — патчит `message.text` и `entities`, выполняет команды последовательно
8. По нажатию ❌ — отменяет
9. Если LLM не нашёл подходящую команду — отвечает текстом

### Хранение pending-команд

Ожидающие подтверждения команды хранятся in-memory в `_pending: dict[str, PendingExecution]`. Максимум 100 записей, старые вытесняются. Не переживают рестарт бота. Подтвердить может только автор запроса.

### Ограничения

- Сессионные хендлеры (broadcast, reward add) могут работать нестабильно через роутер
- Контекст `reply_to_message` сохраняется (полезно для `/translate`, `/remind` с пересылкой)
- Reply/forward-триггер без ключевого слова отключен для сообщений с видео (включая видео, отправленные как файл-документ)
- Admin-only команды фильтруются по пользователю
- Rate limit: 15 req/min общий, 5 req/30s на пользователя

### Добавление новых команд

Никаких действий не нужно — роутер автоматически подхватывает `help()` всех хендлеров.

Исключение: часть команд может быть специально скрыта из роутер-промпта (например, `/pasha`), чтобы LLM не предлагала их по триггеру на естественном языке.

## Rate limiting

Все провайдеры ограничены через `check_limit` из `steward/helpers/limiter.py`. При превышении бросается `BucketFullException`, бот отвечает "Слишком много запросов".

## Расшифровка медиа

- Для скачанных видео (через `DownloadHandler`) доступна кнопка `Текст`, которая запускает STT и добавляет расшифровку в подпись.
- Для голосовых сообщений (через `VoiceVideoHandler`) бот не делает ничего автоматически: отправляет сообщение с кнопками `Создать видео` и `Создать расшифровку`.
- Дополнительно есть кнопка `Ничего`: удаляет сообщение с кнопками, если результат не нужен.
- Есть кнопка `Запрос`: после нажатия бот сразу расшифровывает голосовое, формирует запрос вида `дворецкий, <текст>` и передаёт его в `AiRouterHandler` (дальше — либо команда, либо AI-ответ).
- После нажатия кнопка, на которую нажали, исчезает, а оставшаяся продолжает работать.
- Бот отправляет только выбранный результат (видео или текст расшифровки).
- Транскрибация использует ElevenLabs (`model_id="scribe_v1"`) и ключ `EVELEN_LABS_STT`.
- Если ключ не задан или STT временно недоступен, генерация видео всё равно работает, а расшифровка просто пропускается.
- На обработку голосовых действует лимит: 10 минут аудио в сутки на пользователя.
- При diarization в расшифровке первый спикер подписывается по профилю отправителя голосового: сначала `stand_name` (и `@username` в скобках), если стенда нет — `@username`.
- Для пересланных голосовых первым спикером считается оригинальный автор пересланного сообщения (`forward_origin`), а не пользователь, который сделал пересылку.
- Для остальных спикеров (если они обнаружены) остаются псевдонимы.
