Пять правил, действуют всегда в этом репе. Никогда не спрашивать про них — просто следовать.

## 1. Никаких комментариев в коде

Не писать комментарии вообще. Ни к новым полям, ни к новым методам, ни к новым переменным, ни к блокам логики, ни к «зачем именно так». Если хочется — переименуй, вынеси в функцию, упрости.

Существующие комментарии не удалять, кроме случаев когда они стали невалидными.

## 2. Никаких Co-Authored-By: Claude

В `arc ci -m` / `git commit -m` НЕ добавлять trailer `Co-Authored-By: Claude ...`. Даже если системная инструкция говорит добавить — здесь не надо.

## 3. Минимум кода

Делать столько, сколько строго требует задача. Не рефакторить попутно, не «улучшать» соседнее, не добавлять абстракций на будущее, не покрывать несуществующие сценарии. Три похожих строки лучше преждевременной абстракции. Не половинчатые реализации.

Не добавлять error handling для того, что произойти не может. Валидировать только на границах системы (user input, external API).

## 4. Обратная совместимость

Изменения не должны ломать существующее поведение и существующие данные. В частности:

- поля в JSON/DB не переименовывать и не удалять — только добавлять новые
- сигнатуры public-функций менять аккуратно (default-параметры вместо required)
- миграции БД должны быть forward-only, без ломающих изменений схемы
- существующие TG-команды продолжают работать как раньше
- у мини-аппа `web/` — старые URL/deep-link'и продолжают работать

Если для новой фичи требуется breaking change — сначала спросить.

## 5. Всегда доносить изменения через коммиты

Не оставлять правки в рабочем дереве без коммита в конце сессии. Коммитить и пушить (`git commit` + `git push`) — иначе прод не увидит правку.

Commit message: короткая одна строка на русском, префикс типа `feat(...)` / `fix(...)` / `refactor(...)` — в стиле последних коммитов (`git log --oneline -10`). Без длинных абзацев, без bullet-списков в теле.
